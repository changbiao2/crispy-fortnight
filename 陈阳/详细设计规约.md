## 1. 引言

### 1.1 编写目的

本文档是网上购物网站项目的详细设计规约，旨在为系统开发提供详细的设计指导，包括系统架构、模块划分、接口定义、数据结构设计等内容。

### 1.2 项目背景

本项目是一个B2C模式的网上购物网站，包含前端用户购物系统和后台管理系统两大部分。系统采用前后端分离架构，支持商品展示、购物车管理、订单处理、用户管理、数据统计等核心功能。

### 1.3 项目团队分工

| 成员 | 角色 | 负责模块 |
|------|------|----------|
| 李昶标 | 项目负责人 | 商品展示、查询、购物车 |
| 邓耀华 | 需求分析师 | 用户登录注册、留言、在线客服 |
| 张俊杰 | 架构设计师 | 后台管理、库存、数据统计分析 |
| 陈阳 | 详细设计师 | 详细设计、代码生成、文档整合 |

### 1.4 术语定义

| 术语 | 说明 |
|------|------|
| B2C | Business-to-Consumer，企业对消费者的电子商务模式 |
| RESTful | 一种软件架构风格，基于HTTP协议设计API |
| JWT | JSON Web Token，用于用户身份认证的令牌 |
| DAO | Data Access Object，数据访问对象模式 |

### 2.2 系统分层架构

本系统采用经典的三层架构设计：

#### 2.2.1 表示层（Presentation Layer）
- **前端用户界面**：面向普通用户的购物网站界面
- **后台管理界面**：面向管理员的后台管理系统界面
- **技术选型**：HTML5、CSS3、JavaScript、Vue.js/React

#### 2.2.2 业务逻辑层（Business Logic Layer）
- **控制器层（Controller）**：接收请求、调用服务、返回响应
- **服务层（Service）**：实现业务逻辑、事务管理
- **技术选型**：Spring Boot、Spring MVC

#### 2.2.3 数据访问层（Data Access Layer）
- **DAO层**：数据库访问接口
- **Mapper层**：MyBatis映射文件
- **技术选型**：MyBatis、MySQL

## 4. 泳道图设计

### 4.1 用户购物流程泳道图

**参与角色**：用户、商品系统、购物车系统、订单系统、支付系统

**流程说明**：
1. 用户浏览商品列表
2. 查看商品详情
3. 添加商品到购物车
4. 查看购物车并确认商品
5. 提交订单
6. 选择支付方式并完成支付
7. 系统生成订单并通知用户

**泳道划分**：
- 用户泳道：浏览商品、查看详情、操作购物车、提交订单、支付
- 商品系统泳道：提供商品信息、校验库存
- 购物车系统泳道：添加商品、修改数量、计算总价
- 订单系统泳道：生成订单、更新订单状态
- 支付系统泳道：处理支付请求、返回支付结果

### 4.2 订单处理流程泳道图

**参与角色**：用户、系统、管理员、物流系统

**流程说明**：
1. 用户提交订单
2. 系统校验订单信息
3. 用户完成支付
4. 系统更新订单状态为"待发货"
5. 管理员查看待发货订单
6. 管理员填写物流信息并确认发货
7. 系统更新订单状态为"已发货"
8. 物流系统配送商品
9. 用户确认收货
10. 系统更新订单状态为"已完成"

**泳道划分**：
- 用户泳道：提交订单、支付、确认收货
- 系统泳道：订单校验、状态更新、通知发送
- 管理员泳道：查看订单、处理发货
- 物流系统泳道：配送跟踪、状态同步

### 4.3 库存管理流程泳道图

**参与角色**：管理员、系统、数据库

**流程说明**：
1. 管理员查看库存列表
2. 发现库存不足或需要补货
3. 执行入库操作
4. 系统更新库存数量
5. 系统记录入库日志
6. 库存数据同步到数据库
7. 系统检查是否触发库存预警
8. 如触发预警，通知管理员

**泳道划分**：
- 管理员泳道：查看库存、执行入库/出库
- 系统泳道：库存计算、预警检查、日志记录
- 数据库泳道：数据持久化、事务管理

## 5. 状态图设计

### 5.1 订单状态图

**状态列表**：
- 待支付（Pending Payment）
- 已支付（Paid）
- 待发货（Pending Shipment）
- 已发货（Shipped）
- 已完成（Completed）
- 已取消（Cancelled）

**状态转换**：
```
创建订单 → 待支付
待支付 → 已支付（用户完成支付）
待支付 → 已取消（用户取消或超时未支付）
已支付 → 待发货
待发货 → 已发货（管理员发货）
已发货 → 已完成（用户确认收货或自动确认）
已发货 → 已取消（退货处理）
```

### 5.2 商品状态图

**状态列表**：
- 草稿（Draft）
- 已上架（On Sale）
- 已下架（Off Sale）

**状态转换**：
```
创建商品 → 草稿
草稿 → 已上架（管理员上架操作）
已上架 → 已下架（管理员下架操作）
已下架 → 已上架（重新上架）
```

### 5.3 用户状态图

**状态列表**：
- 未激活（Inactive）
- 正常（Active）
- 禁用（Disabled）

**状态转换**：
```
注册 → 未激活
未激活 → 正常（完成邮箱/手机验证）
正常 → 禁用（管理员禁用）
禁用 → 正常（管理员启用）
```

### 6.2 设计模式应用

#### 6.2.1 工厂模式（Factory Pattern）

**应用场景**：商品、订单、用户等对象的创建

**设计说明**：
- 定义产品接口（ProductInterface）
- 实现具体产品类（Product、Order、User）
- 创建工厂类（EntityFactory）负责对象实例化
- 通过工厂方法统一创建对象，隐藏创建细节

#### 6.2.2 外观模式（Facade Pattern）

**应用场景**：为复杂的子系统提供统一入口

**设计说明**：
- 创建外观类（ShoppingFacade）
- 封装商品、购物车、订单等子系统的复杂操作
- 对外提供简洁的接口（如：purchase()）
- 降低客户端与子系统的耦合度

#### 6.2.3 单例模式（Singleton Pattern）

**应用场景**：配置管理、数据库连接池

**设计说明**：
- 配置管理器（ConfigManager）：全局唯一实例
- 数据库连接池（ConnectionPool）：统一管理连接
- 缓存管理器（CacheManager）：全局缓存实例

#### 6.2.4 DAO模式（Data Access Object）

**应用场景**：数据访问层统一接口

**设计说明**：
- 定义DAO接口（BaseDao<T>）：增删改查基本操作
- 实现具体DAO类（ProductDao、UserDao、OrderDao）
- 通过DAO层隔离业务逻辑与数据访问
- 便于切换数据源和进行单元测试

## 7. 接口设计汇总

### 7.1 接口设计原则

1. **RESTful风格**：采用HTTP方法（GET/POST/PUT/DELETE）表示操作
2. **统一响应格式**：所有接口返回统一格式的JSON数据
3. **版本控制**：API路径包含版本号（如：/api/v1/products）
4. **认证授权**：敏感接口使用JWT Token进行身份验证

### 7.2 统一响应格式

```json
{
  "code": 200,
  "message": "操作成功",
  "data": { ... }
}
```

**状态码说明**：

| 状态码 | 说明 |
|--------|------|
| 200 | 操作成功 |
| 400 | 请求参数错误 |
| 401 | 未登录或token过期 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 500 | 服务器内部错误 |

## 8. 数据库设计汇总

### 8.1 数据库设计原则

1. **第三范式（3NF）**：消除传递依赖，确保数据一致性
2. **统一命名规范**：表名使用小写下划线，字段名使用驼峰命名
3. **主键策略**：使用自增ID或UUID作为主键
4. **索引设计**：为常用查询字段添加索引
5. **外键约束**：保持数据完整性
